# バトルバランス調整ループ

このプロジェクトでは、バトルの強さを手触りだけで調整しないために、ヘッドレスの自動戦闘を使う。

目的は次の状態に近づけること。

- CPU同士では敵がやや有利
- プレイヤー相当の操作方針では自軍が不利を覆せる
- 突破だけ、包囲だけが常に最適にならない

## 実行方法

```bash
npm run balance
```

実行すると、Viteでバランスランナーを一時的に `dist-balance/` へbundleし、Node上で `BattleSimulation` を複数シード実行する。デフォルトは `1001,1002,1003,1004` の4シード、最大120秒。

結果は以下へ出力する。

- `balance-reports/latest.md`
- `balance-reports/latest.json`

`dist-balance/` は一時生成物なのでGit管理しない。

## 方針

| 方針 | 意味 |
| --- | --- |
| `cpu-basic` | 自軍も雑なCPUとして操作する基準。ここでは敵がやや有利でよい |
| `player-adaptive` | 隙間なら突破、過密なら包囲、包囲中なら維持する上手いプレイヤー相当 |
| `breakthrough-only` | 密集突破だけを繰り返す一強チェック |
| `envelop-only` | 展開包囲だけを繰り返す一強チェック |

## 目標値

| 条件 | 目標 |
| --- | --- |
| `cpu-basic` | 敵スコア率 55〜65% |
| `player-adaptive` | 自軍スコア率 55〜70% |
| `breakthrough-only` | `player-adaptive` を大きく超えない |
| `envelop-only` | `player-adaptive` を大きく超えない |

スコア率は、勝ちを1、負けを0、引き分けや時間切れを0.5として集計する。

## 調整対象

優先して触る値は以下。

- 敵AIのミス率、判断頻度
- 初期士気、結束、疲労、靱性
- 包囲による士気・疲労・指揮遅延
- 突破による疲労、結束低下
- `gapRisk` と `crowding`
- 敗走閾値

調整は一度に大きく変えず、同じシードで再実行して差分を見る。

## 現在の基準結果

2026-06-22時点の4シード検証では、以下の結果を基準にしている。

| 方針 | スコア率 | 判定 |
| --- | ---: | --- |
| `cpu-basic` 敵スコア率 | 63% | OK |
| `player-adaptive` 自軍スコア率 | 63% | OK |
| `breakthrough-only` 自軍スコア率 | 38% | OK |
| `envelop-only` 自軍スコア率 | 63% | OK |

この状態では、CPU同士では敵がやや有利だが、適応的なプレイヤー方針では自軍が覆せる。突破だけは敵に潰されやすく、包囲だけは勝てる場合もあるが、敵の一点突破や薄い包囲線の士気コストで安定最適にはならない。
